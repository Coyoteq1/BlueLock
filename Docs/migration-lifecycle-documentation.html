Purpose: Handles player kit requests and manages the kit distribution workflow in the ECS architecture.

Description: KitRequestSystem processes player requests for kits, validates them, and coordinates with other services to fulfill the request.

Dependencies
EndGameKitSystem, VrisingItemApi, EntityManager
Input Parameters

  • KitName (string)
  • RequestContext (RequestContext)
Return Values
RequestResult (enum: Success, Failed, Pending)

Key Methods

  • ProcessRequestAsync(request) - Process a kit request asynchronously
  • ValidateRequest(request) - Validate request parameters
  • ExecuteRequest(request) - Execute the validated request

Error Handling

  • InvalidKitNameException - Thrown when kit name is not found

⚠️ Edge Cases Documentation

Edge Case 1: Long GUID Overflow Handling

Scenario: Converting legacy int GUIDs to long format when values exceed int.MaxValue

Impact: Data corruption if not handled properly

Mitigation:

  • Check for overflow before conversion
  • Implement fallback to new GUID generation
  • Log warning when legacy data exceeds limits
try {
    long longGuid = PrefabGuidConverter.ConvertToLong(intGuid);
}
catch (OverflowException) {
    // Generate new GUID for the item
    var newGuid = PrefabGuidConverter.GenerateNewGuid();
    LogWarning($"Legacy GUID overflow, generated new: {newGuid}");
}
Edge Case 2: Entity Destruction During Processing Handling

Scenario: Entity being processed is destroyed by another system mid-operation

Impact: Null reference exceptions, data inconsistency

Mitigation:

  • Check EntityManager.Exists before all operations
  • Use EntityCommandBuffer for deferred operations
  • Implement transaction-like atomic operations
if (!EntityManager.Exists(entity)) {
    LogDebug($"Entity {entity} no longer exists, skipping processing");
    return;
}
Edge Case 3: Config File Corruption Handling

Scenario: Configuration file is corrupted or contains invalid JSON

Impact: System fails to load, plugin may crash

Mitigation:

  • Implement try-catch around config loading
  • Backup config before modification
  • Fall back to default configuration
  • Provide clear error messages to user
Edge Case 4: Race Condition in Service Initialization Handling

Scenario: Multiple services try to initialize concurrently causing dependency issues

Impact: Service dependencies not available, initialization failures

Mitigation:

  • Implement proper initialization order in ServiceManager
  • Use async initialization with dependency checking
  • Log initialization progress and failures
Edge Case 5: Player Disconnect During Kit Distribution Handling

Scenario: Player disconnects while kit distribution is in progress

Impact: Partial inventory changes, orphaned items

Mitigation:

  • Check player online status before distribution
  • Use transaction-like approach (all or nothing)
  • Clean up partial changes on disconnect

🚨 Error Scenarios & Rollback Procedures

Error Scenario 1: PrefabGUID Conversion Failure Critical

Error: PrefabGUID conversion from int to long fails due to invalid data

Severity: Critical

Impact: Item spawning fails, game may crash

Rollback Procedure:
  1. Log error with full stack trace and GUID value
  2. Fall back to using legacy int conversion if available
  3. If conversion completely fails, use placeholder GUID
  4. Continue operation with degraded functionality
  5. Alert admin via configured notification channel
Error Scenario 2: Service Initialization Failure High

Error: Core service fails to initialize (e.g., EndGameKitSystem)

Severity: High

Impact: Feature unavailable, other services may cascade

Rollback Procedure:
  1. Stop all dependent services
  2. Log detailed initialization failure
  3. Attempt secondary initialization method
  4. If still failing, disable feature gracefully
  5. Continue plugin load without failing service
Error Scenario 3: ECS Query Returns Invalid Entity Medium

Error: ECS query returns entity that doesn't exist or has invalid components

Severity: Medium

Impact: Processing errors, potential memory leaks

Rollback Procedure:
  1. Verify entity existence with EntityManager.Exists
  2. Skip processing if entity is invalid
  3. Log warning with entity ID
  4. Continue processing other entities
  5. Schedule entity cleanup job
Error Scenario 4: Config Reload Failure Medium

Error: Runtime config reload fails due to invalid JSON or permissions

Severity: Medium

Impact: Config changes not applied, potential state inconsistency

Rollback Procedure:
  1. Keep existing valid configuration
  2. Log reload failure with file path
  3. Create backup of invalid config file
  4. Continue with last known good config
  5. Notify admin of config error
Error Scenario 5: Multiplayer Desync Critical

Error: State desynchronizes between server and clients during gameplay

Severity: Critical

Impact: Gameplay issues, player frustration, potential crashes

Rollback Procedure:
  1. Detect desync via checksum validation
  2. Force resync of affected player's state
  3. Log desync event with player and state details
  4. Continue with corrected state
  5. Consider full resync if issue persists
